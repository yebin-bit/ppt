# HTML 선행 디자인 승인 루프 (case 3·4)

## 왜
PPT 안에서 디자인을 반복하면 느리다. HTML/CSS는 색·레이아웃·간격을 즉시 보고 고치기 빠르다. 그래서
**HTML은 "디자인을 빠르게 확정·승인받는 도구"로만** 쓰고, 확정되면 그 결정을 design.md에 고정한 뒤
**최종 pptx는 네이티브로 재구축**한다. HTML을 그대로 pptx로 통째 변환(html2pptx)하지 않는 이유: 편집성·
재사용·충실도 트레이드오프가 있고, 우리의 목표는 "복제·편집 가능한 덱"이기 때문이다.

## 절차
1. **슬라이드 캔버스 HTML 작성** — 한 장을 한 `.slide` div로. 16:9면 1280×720, 4:3면 1024×768(또는
   A4 가로 1123×794) 등 목표 비율에 맞춰 고정 px. CSS 변수로 색 토큰을 선언해 한곳에서 제어.
2. **사용자에게 보여주고 승인** — 색/레이아웃/타이포를 빠르게 반복. (이 레포의 데스크탑 작업물처럼.)
3. **design.md에 고정** — 확정된 색 토큰·폰트·타입스케일·레이아웃 패턴을 design.md에 옮긴다.
   CSS 변수 → design.md 색 토큰, px 간격 → 0.3"/0.5" 그리드로 환산.
4. **네이티브 재구축** — 3단계 결정트리로:
   - 브랜드 시스템·여러 마스터가 필요 → `pptx-theme-engine` (tokens.json은 design.md에서 파생).
   - 차트 든 반복 양식 → 대표 슬라이드 1장을 네이티브로 만든 뒤 `pptx-chart-template-filler`로 복제.

## 주의
- HTML 픽셀을 1:1로 옮기려 하지 말 것 — 비율·위계·간격을 옮긴다(근사 승인용이었음).
- 공식 pptx 디자인 가이드 준수: **언더라인/장식 사선 남발 금지**(AI 티), 텍스트박스 margin 0, 대비 확보.
- 폰트는 PowerPoint에 실제로 있는 폰트로 매핑(HTML 웹폰트 ≠ PPT 폰트). QA 신뢰도 위해 안전폰트 권장.
